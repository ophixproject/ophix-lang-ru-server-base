Каждый сервер Ophix ведёт журнал событий доступа клиентов к артефактам. Журнал аудита встроен в `ophix-server-base` и не требует дополнительной настройки — он активен на всех серверах доменов (учётные данные, конфигурации, сертификаты и любые будущие домены).

---

## Что записывается в журнал

Каждый раз, когда клиент успешно читает или записывает артефакт, создаётся запись `AccessLog` со следующими полями:

| Поле | Описание |
| --- | --- |
| `client` | Клиент, выполнивший запрос |
| `host` | Хост, на котором выполняется клиент |
| `operation` | `GET` (чтение) или `POST` (запись/создание) |
| `artifact_type` | Имя класса модели — `Credential`, `CertBundle` и т. д. |
| `artifact_name` | Имя артефакта на момент доступа (снимок — сохраняется при переименовании и удалении) |
| `artifact_id` | Первичный ключ артефакта на момент доступа (сохраняется при удалении артефакта) |
| `timestamp` | Время запроса (устанавливается при создании события, не при записи в базу данных) |

Записываются только успешные операции. Запросы, отклонённые при проверке аутентификации или прав доступа, не попадают в журнал аудита (они фиксируются в журнале приложения через стандартное логирование Django).

---

## Просмотр журнала аудита

Журнал аудита доступен в административной панели Django в разделе **Ophix Core → Журналы доступа**.

Список поддерживает фильтрацию по:

- Клиенту
- Хосту
- Операции (GET / POST)
- Типу артефакта
- Дате (навигация по иерархии дат)

Журнал доступен только для чтения — записи нельзя добавлять, изменять или удалять через административную панель.

---

## Производительность

Запись аудита является неблокирующей по своей конструкции. Представления домена помещают события в очередь в памяти и немедленно возвращают управление — поток запроса никогда не ожидает записи в базу данных. Фоновый поток-демон опустошает очередь и записывает пакеты через один вызов `bulk_create()`.

Если при экстремальной нагрузке очередь переполнится, избыточные события будут отброшены, а в журнал приложения будет записано предупреждение. Аудит никогда не создаёт обратного давления на основной сервис обработки запросов.

## Настройки

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `AUDIT_BATCH_SIZE` | `50` | Количество событий, накапливаемых перед записью пакета в базу данных. |
| `AUDIT_FLUSH_INTERVAL` | `5` | Максимальное время ожидания (в секундах) перед сбросом неполного пакета. Гарантирует, что события не задерживаются при низком трафике. |

Эти параметры можно задать в `.env`. Для большинства развёртываний значения по умолчанию подходят.

## Удаление старых записей

Записи журнала аудита накапливаются бесконечно, если их не очищать. Используйте команду управления `prune_access_log` для удаления старых записей:

```bash
# Удалить записи старше 90 дней (по умолчанию)
ophix-manage prune_access_log

# Удалить записи старше 30 дней
ophix-manage prune_access_log --days 30

# Предварительный просмотр количества записей к удалению
ophix-manage prune_access_log --dry-run
ophix-manage prune_access_log --days 30 --dry-run
```

Выполняйте эту команду по расписанию — типичный вариант: ежедневное или еженедельное задание cron:

```cron
0 3 * * 0   ophixuser  /path/to/venv/bin/ophix-manage prune_access_log --days 90
```

На рабочем сервере всегда выполняйте `--dry-run` перед изменением периода хранения, чтобы убедиться в корректности результата.
