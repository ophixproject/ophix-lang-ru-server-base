Каждый Ophix-клиент первого уровня следует одному и тому же процессу инициализации. В примерах ниже используется `cred-client` — замените его на `cert-client`, `conf-client` и т. д. в зависимости от домена.

---

## Установка

```bash
pip install ophix-cred-client
```

---

## Инициализация в один шаг

```bash
cred-client quickstart https://credserver.internal мой-клиент
```

Эта команда:

1. Записывает URL сервера в `.cred.env`
2. Загружает CA-сертификат сервера и сохраняет его локально
3. Регистрирует клиент на сервере и сохраняет API-токен

На этом всё — клиент готов к работе.

---

## Пошаговая инициализация

Если вы предпочитаете контролировать каждый шаг отдельно:

```bash
cred-client set server https://credserver.internal
cred-client download ca-cert
cred-client register мой-клиент
```

---

## Проверка конфигурации

```bash
cred-client doctor
```

Отображает состояние конфигурации: URL сервера, CA-сертификат, токен и доступность соединения.

---

## Файл конфигурации

Конфигурация хранится в `.cred.env` (в текущем каталоге или в `~`). Пример:

```ini
CREDSERVER_URL=https://credserver.internal
CREDSERVER_CA_CERT=/home/user/.cred-ca.pem
CREDSERVER_API_TOKEN=<64-символьный hex-токен>
```

---

## Ротация токена

Токены следует обновлять регулярно. Ротация генерирует новый токен, проверяет его на сервере, а затем обновляет файл конфигурации:

```bash
cred-client rotate-token
```

Для планирования ротации (cron):

```cron
0 2 * * 1   user  /path/to/venv/bin/cred-client rotate-token
```

Если сервер недоступен, ротация завершается с явной ошибкой — никогда не подавляйте её вывод.

---

## Предоставление доступа к артефактам

По умолчанию клиенты не имеют доступа ни к каким артефактам. Администратор должен явно создать связь через административную панель Django:

1. Открыть страницу сведений о **Клиенте** или **Артефакте** в панели администратора
2. Использовать встроенную форму для создания связи
3. Установить флажок **Enabled** на записи связи

После активации связи клиент сможет получать артефакт.
